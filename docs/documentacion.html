DiveComputer — Documentación

Ordenador de buceo de código abierto basado en ESP32. Muestra rumbo compensado por inclinación, profundidad y tiempo de inmersión en un display TFT de 1.8". Incluye portal WiFi para calibración del magnetómetro sin necesidad de herramientas adicionales.

Alimentación

El ESP32-C3-DevKitC-02 se alimenta por USB-C a 5 V y dispone de un regulador interno que genera 3.3 V para todos los periféricos. No se necesita regulador externo.

⚠ Tensiones lógicas
El ESP32-C3 opera a 3.3 V. El ST7735, HMC5883L y LSM303 también funcionan a 3.3 V. No conectes directamente señales de 5 V a ninguno de estos pines.

Requisitos

  • Driver USB-CDC — en Windows puede requerir reinicio tras conectar por primera vez
  • Python 3.8+ (gestionado automáticamente por PlatformIO)

El firmware habilita ARDUINO_USB_CDC_ON_BOOT=1. El monitor serie aparece como puerto virtual tras la enumeración USB; si no aparece, desconecta y reconecta el cable.

Arquitectura del firmware

El código principal reside en src/main.cpp; el motor de descompresión en src/deco.h. El ESP32-S3 tiene dos núcleos Xtensa LX7 y el firmware los aprovecha:

NúcleoTarea FreeRTOSResponsabilidadFrecuencia
Core 1 (Arduino default) loop() Brújula, acelerómetro, TFT, WiFi, SD ~33 Hz (delay 30 ms)
Core 0 depthTask Profundidad + algoritmo Bühlmann ZHL-16C 10 Hz (100 ms)

Orden de inicialización (setup())

  1. Serial — USB CDC a 115 200 baud. Espera hasta 2 s a la enumeración USB.
  2. TFT — SPI.begin(4,−1,3) → tft.initR(INITR_BLACKTAB) → rotación landscape → splash screen.
  3. I²C — Wire.begin(I2C_SDA, I2C_SCL) con los pines definidos en platformio.ini.
  4. HMC5883L — mag.begin(). Bucle infinito si el sensor no responde.
  5. LSM303 — accel.begin(0x18). SA0=GND → dirección 0x18.
  6. Preferencias — Carga calibración almacenada en flash NVS (namespace "compass").
  7. WiFi AP — Punto de acceso Brujula-Calib en canal 1 + servidor HTTP en puerto 80.
  8. depthTask — Crea el mutex de datos compartidos y lanza depthTask en Core 0.

Core 1 — loop() (cada 30 ms)

dnsServer.processNextRequest();
server.handleClient();             // portal WiFi

accel.getEvent(&accEv);
mag.getEvent(&magEv);

if (calState.collecting)
    updateCalibrationSamples(magEv);

float heading = calculateHeading(magEv, accEv);

// Leer datos producidos por Core 0 (mutex protegido)
xSemaphoreTake(g_dataMutex, 5 ms);
depth = g_depth;  deco = g_deco;
xSemaphoreGive(g_dataMutex);

drawHeadingStrip(heading);
drawDepth(depth);
drawDiveData(g_tempC, deco);    // NDL / techo de deco
if (hdgChanged) drawCompassBar(heading);
logToSD(depth, g_tempC, heading, elapsed);

delay(30);

Core 0 — depthTask (cada 100 ms)

// Inicialización (una sola vez)
DecoEngine engine;
engine.init();    // tejidos a saturación de superficie (aire, 1 bar)

for (;;) {
    depth = readPressureSensor();   // TODO: sensor real (MS5837, etc.)
    engine.update(depth, 0.1f);     // integra Haldane 100 ms
    deco  = engine.calculate(depth); // NDL analítico + techo

    xSemaphoreTake(g_dataMutex, portMAX_DELAY);
    g_depth = depth;
    g_deco  = deco;
    xSemaphoreGive(g_dataMutex);

    vTaskDelay(pdMS_TO_TICKS(100));
}
ℹ Sincronización entre cores
g_depth y g_deco son escritas por Core 0 y leídas por Core 1. El acceso está protegido por un mutex FreeRTOS (g_dataMutex). Core 1 usa un timeout de 5 ms para no bloquear el render si Core 0 está en mitad de un cálculo.

Algoritmo de rumbo compensado

La función calculateHeading() calcula el rumbo magnético corregiendo la inclinación de la placa mediante los datos del acelerómetro.

Remapeo de ejes (montaje físico)

El sensor está montado con X hacia abajo, Y hacia la izquierda, Z hacia adelante. Se remapea al marco de la fórmula (X=adelante, Y=izquierda, Z=arriba): 

// Acelerómetro → marco fórmula
ax = -accEvent.acceleration.z;
ay =  accEvent.acceleration.y;
az = -accEvent.acceleration.x;

// Magnetómetro + calibración → marco fórmula
mx = -(magEvent.magnetic.z - cal.oz) * cal.sz;
my =  (magEvent.magnetic.y - cal.oy) * cal.sy;
mz = -(magEvent.magnetic.x - cal.ox) * cal.sx;

Compensación de inclinación (tilt compensation)

// Pitch y roll del acelerómetro
pitch = atan2(-ax, sqrt(ay*ay + az*az));
roll  = atan2(ay, az);

// Componentes horizontales del campo magnético
Xh = mx*cos(pitch) + mz*sin(pitch);
Yh = mx*sin(roll)*sin(pitch) + my*cos(roll) - mz*sin(roll)*cos(pitch);

// Rumbo + declinación magnética (Alcalá la Real ≈ −0.63°)
heading = atan2(Yh, Xh) + (-0.011);   // rad
heading = fmod(heading + 2*PI, 2*PI) * 180.0 / PI;  // → 0..360°
ℹ Declinación magnética
El valor −0.011 rad (≈ −0.63°) está ajustado para Alcalá la Real, España. Para otra ubicación, calcula tu valor en NOAA Magnetic Field Calculator y actualiza la constante declinationAngle en calculateHeading().

Dual-core FreeRTOS

El ESP32-S3 integra dos núcleos Xtensa LX7 que pueden ejecutar tareas FreeRTOS en paralelo real. El firmware divide el trabajo de forma que el rendering y las comunicaciones nunca bloquean el cálculo de descompresión, y viceversa.

Core 1 — Arduino loop

~33 Hz

Brújula · acelerómetro · TFT · WiFi · DNS · SD logging

Core 0 — depthTask

10 Hz

Sensor de profundidad · Bühlmann ZHL-16C · NDL · techo

Sincronización

Mutex

FreeRTOS xSemaphoreCreateMutex() protege g_depth y g_deco

Stack depthTask

4 KB

Ampliar a 8 KB cuando se añadan tablas de deco Bühlmann completas

Creación de la tarea

// En setup(), tras inicializar WiFi:
g_dataMutex = xSemaphoreCreateMutex();

xTaskCreatePinnedToCore(
    depthTask,   // función de la tarea
    "depthTask", // nombre (para depuración)
    4096,        // stack en bytes
    nullptr,     // parámetro
    1,           // prioridad (baja, para no interferir con WiFi stack)
    nullptr,     // handle (no necesario)
    0            // Core 0
);
⚠ I²C no es thread-safe
El bus I²C (Wire) lo acceden el magnetómetro y el acelerómetro, ambos desde Core 1. El futuro sensor de presión en Core 0 debe usar una instancia separada (Wire1) o un mutex adicional para evitar colisiones en el bus.

Algoritmo de descompresión — Bühlmann ZHL-16C

Implementado en src/deco.h como clase autocontenida sin asignación dinámica de memoria. Referencia: A.A. Bühlmann, Tauchmedizin 5ª ed. y tabla publicada en Wikipedia ZH-L16C.

⚠ Aviso de seguridad
Esta implementación es educativa y está en desarrollo activo. No usar como única fuente de información de seguridad en inmersiones reales. Siempre llevar tablas de buceo y formación adecuada.

Modelo de compartimentos de tejido

El algoritmo modela el cuerpo humano como 16 compartimentos paralelos, cada uno con una velocidad distinta de absorción y eliminación del nitrógeno disuelto. Los compartimentos rápidos representan tejidos muy vascularizados (sangre, cerebro); los lentos representan cartílagos y huesos.

Ecuación de Haldane — carga de gas

Cada compartimento sigue la ecuación de Haldane. Para un intervalo Δt a profundidad constante: 

// Presión ambiente y alveolar
P_amb = 1.0 + depth_m / 10.0          // bar (agua salada simplificada)
P_alv = (P_amb - 0.0627) × 0.7902    // presión alveolar N₂ (vapor agua 37°C)

// Integración Haldane (implementado en DecoEngine::update)
k     = ln(2) / t½                       // constante de tiempo del compartimento
P_t  += (P_alv - P_t) × (1 - e^(-k×Δt)) // nueva carga en el tejido

A mayor profundidad, P_alv sube y los tejidos se cargan de N₂. Al ascender, P_alv baja y los tejidos eliminan N₂. Los compartimentos rápidos (T½ pequeño) reaccionan en minutos; los lentos tardan horas.

Valor M — límite de saturación

Si se asciende directamente a superficie, cada compartimento tiene un límite máximo de N₂ disuelto que no debe superarse para evitar formación de burbujas (enfermedad descompresiva). Este límite depende de la presión ambiente: 

M_i = a_i + P_amb / b_i

Si la carga del tejido P_t > M_i, el buceador no puede ascender directamente a esa presión: necesita una parada de descompresión.

Factor de gradiente (GF)

El factor de gradiente GF ∈ [0, 1] añade un margen de seguridad conservador reduciendo el gradiente efectivo permitido entre tejido y ambiente: 

// Límite con GF en superficie (P_amb = 1 bar)
M_surf_gf = 1 + GF × (a + 1/b - 1)

// Techo con GF (presión mínima segura de ascenso)
P_ceil = (P_t - GF × a) / (1 + GF × (1/b - 1))
GFComportamientoUso típico
1.00M-value puro de BühlmannMáxima permisividad (no recomendado)
0.8585% del gradiente hacia M-valueRecreativo moderado (nuestro valor)
0.7070% del gradienteTécnico conservador
0.30 / 0.85GF_low / GF_high variablesGF interpolado por profundidad (futuro)

Cálculo del NDL (No-Decompression Limit)

El NDL es el tiempo que puede permanecer el buceador a la profundidad actual antes de que algún compartimento supere su límite en superficie. Se calcula analíticamente invirtiendo la ecuación de Haldane — sin bucle de simulación: 

// Para cada compartimento i, si P_alv > P_t (cargando):
ratio = (M_surf_gf - P_t) / (P_alv - P_t)

if (ratio < 1.0):
    t_NDL_i = -ln(1 - ratio) / k    // tiempo hasta M_surf_gf (min)
else:
    t_NDL_i = ∞                       // tejido satura por debajo del límite

// NDL global = mínimo sobre todos los compartimentos
NDL = min(t_NDL_1, t_NDL_2, ..., t_NDL_16)

Cálculo del techo de ascenso

Cuando el buceador está en deco, el techo es la profundidad mínima a la que puede ascender sin que ningún compartimento supere su M-value con GF: 

// Para cada compartimento i:
P_ceil_i = (P_t - GF × a_i) / (1 + GF × (1/b_i - 1))

// Techo global = compartimento más restrictivo
P_ceil = max(P_ceil_1, ..., P_ceil_16)
depth_ceil = (P_ceil - 1.0) × 10.0   // metros

Implementación — src/deco.h

La clase DecoEngine encapsula el estado de los 16 compartimentos y expone tres métodos. No usa new, malloc, ni ningún contenedor de la STL, por lo que es segura para FreeRTOS con stack limitado. 

struct DecoResult {
  float ceiling_m;  // techo mínimo de ascenso (m), 0 si no hay deco
  float ndl_min;    // NDL restante (min), 0 si ya está en deco
  bool  in_deco;
};

class DecoEngine {
public:
  float pN2[16];         // estado de los tejidos (bar)

  void init();              // tejidos → saturación de superficie
  void update(float depth_m, float dt_s);   // Haldane Δt
  DecoResult calculate(float depth_m) const; // NDL + techo
};

Ejemplo numérico — 20 m, 30 minutos

Partiendo de tejidos a saturación de superficie (aire, 1 bar):

Condiciones iniciales:
  P_amb  = 1.0 + 20/10 = 3.0 bar
  P_alv  = (3.0 - 0.0627) × 0.7902 = 2.321 bar
  P_t_0  = (1.0 - 0.0627) × 0.7902 = 0.745 bar  (superficie)

Compartimento 5 (T½ = 27 min, a = 0.62, b = 0.8126) tras 30 min:
  k      = ln(2) / 27 = 0.02567 min⁻¹
  P_t    = 0.745 + (2.321 - 0.745) × (1 - e^(-0.02567×30))
         = 0.745 + 1.576 × 0.536 = 1.590 bar

Límite en superficie con GF = 0.85:
  M_surf = 1 + 0.85 × (0.62 + 1/0.8126 - 1)
         = 1 + 0.85 × 0.851 = 1.723 bar  ← P_t < M_surf ✓

NDL analítico para este compartimento:
  ratio  = (1.723 - 1.590) / (2.321 - 1.590) = 0.182
  t_NDL  = -ln(1 - 0.182) / 0.02567 = 7.8 min

Compartimento más restrictivo a 20 m a los 30 min → NDL ≈ 8 min
✓ NDL real a 20 m según tablas PADI
Las tablas PADI indican un NDL de 45 min a 18 m y 20 min a 22 m, interpolando ≈ 30 min a 20 m para un buceador virgen. El valor Bühlmann con GF=0.85 es más conservador porque considera todos los compartimentos simultáneamente, no solo el más limitante.

Portal de calibración WiFi

Tras arrancar, el ESP32 levanta un punto de acceso WiFi abierto. Conéctate desde cualquier dispositivo y abre el portal para calibrar el magnetómetro sin desmontar ni reconectar nada.

Procedimiento de calibración

  1. Conecta al punto de acceso — busca Brujula-Calib en la lista WiFi de tu móvil y conéctate.
  2. Abre el portal — navega a http://192.168.4.1 en el navegador. Verás el rumbo actual en tiempo real.
  3. Inicia la calibración — pulsa "Iniciar calibración". El estado cambia a calibrating.
  4. Mueve el dispositivo — dibuja un "8" en el aire durante 20–30 s, cubriendo todas las orientaciones posibles (inclina, gira, voltea). Observa cómo aumentan las muestras y los valores min/max.
  5. Detén y guarda — pulsa "Detener y guardar". El firmware calcula offsets hard-iron y escalas soft-iron automáticamente y los almacena en flash NVS.
✓ ¿Cuántas muestras son suficientes?
El firmware solo acepta la calibración si se han recogido más de 50 muestras. Para resultados óptimos, apunta a 200–500 muestras rotando en múltiples ejes.

Algoritmo de calibración hard-iron + soft-iron

// Hard-iron: offset = centro del elipsoide
cal.ox = (maxX + minX) / 2;
cal.oy = (maxY + minY) / 2;
cal.oz = (maxZ + minZ) / 2;

// Soft-iron: escalar cada eje al radio medio
cx  = (maxX - minX) / 2;
avg = (cx + cy + cz) / 3;
cal.sx = avg / cx;   // idem para sy, sz
